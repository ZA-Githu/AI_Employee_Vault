## 1. Purpose & Vision

Gold Tier extends the Silver approval layer into a **fully integrated, cross-domain AI employee**. Whereas Silver introduced structured planning and human approval workflows, Gold adds autonomous multi-step execution, external social media publishing, business accounting audits, CEO briefing generation, and robust error recovery — all governed by the Ralph Wiggum Loop.

**Core promise:** The agent operates as a full-time AI employee across Personal and Business domains. It posts content, monitors platforms, audits finances, generates CEO briefings, and recovers from failures — all traceable, all logged, all under human governance.

**Skill location:** All Gold Agent Skills are defined as `.md` files in the `skills/` folder at vault root. This is the canonical location for all tiers (Bronze, Silver, and Gold). Never place skill files in `.claude/skills/`.

## 4. Gold Folder Rules

### 4.1 Accounting/

**Purpose:** Financial records, expense logs, income summaries, and weekly accounting audit reports.

**Rules:**
- Files created by the agent during weekly audit cycles.
- Naming: `YYYY-MM-DD — <Report-Title>.md`
- Sub-folders allowed: `Accounting/2026-02/`
- Never delete accounting records — append or archive only.
- All monetary values must include currency (PKR, USD, etc.).

**Required frontmatter:**
```yaml
---
title: ""
type: accounting-report
period: "YYYY-MM-DD to YYYY-MM-DD"
status: draft | reviewed | approved
created: YYYY-MM-DD
agent_generated: true
tags: [accounting, audit, gold]
---
```

### 4.2 Briefings/

**Purpose:** Weekly CEO Briefings — a structured executive summary of all business and personal activity.

**Rules:**
- One briefing per week, generated every Monday (or on demand).
- Naming: `YYYY-WW — CEO-Briefing.md` (ISO week number)
- The briefing is **auto-generated** by the agent but marked `status: pending-review` until the human acknowledges it.
- Never overwrite a completed briefing — create a new revision with suffix `-r2`, `-r3` etc.
- Sub-folders by year: `Briefings/2026/`

**Required frontmatter:**
```yaml
---
title: "CEO Briefing — Week WW, YYYY"
type: ceo-briefing
week: WW
year: YYYY
period: "YYYY-MM-DD to YYYY-MM-DD"
status: pending-review
generated_by: claude
generated_at: YYYY-MM-DD HH:MM
tags: [briefing, ceo, weekly, gold]
---
```

**CEO Briefing Sections (required in this order):**
1. Executive Summary (3–5 bullet points)
2. Tasks Completed This Week
3. Social Media Activity Summary
4. Accounting / Financial Summary
5. Pending Approvals & Blockers
6. Upcoming Deadlines
7. Agent Recommendations

### 4.3 Social_Summaries/

**Purpose:** Weekly summaries of all social media activity (Facebook, Instagram, Twitter/X, LinkedIn).

**Rules:**
- One summary per week per platform, or one combined weekly summary.
- Naming: `YYYY-WW — Social-Summary.md`
- Auto-generated after each posting cycle or weekly.
- Sub-folders by year: `Social_Summaries/2026/`

**Required frontmatter:**
```yaml
---
title: "Social Summary — Week WW, YYYY"
type: social-summary
week: WW
year: YYYY
platforms: [facebook, instagram, twitter, linkedin]
status: complete
generated_by: claude
tags: [social, summary, weekly, gold]
---
```

## 5. Ralph Wiggum Loop (Autonomous Multi-Step Execution)

The Ralph Wiggum Loop is Gold Tier's autonomous execution engine. Named after the character who "just does things" — the agent keeps acting until the task is fully complete or it hits a hard stop condition.

### 5.1 Loop Definition

```
START: Agent receives approved task or scheduled trigger
  ↓
PLAN: Agent decomposes task into atomic steps
  ↓
EXECUTE: Agent runs step 1
  ↓
CHECK: Did it succeed?
  ├─ YES → log success → move to step 2
  └─ NO  → run error recovery → retry once → if still failing → ESCALATE
  ↓
REPEAT until all steps complete or ESCALATE triggered
  ↓
REPORT: Write completion note → update CEO Briefing queue → log LOOP_COMPLETE
```

### 5.2 Hard Stop Conditions

The loop **must stop immediately** if any of these are true:

| Condition | Action |
|-----------|--------|
| Step fails after 1 retry | Escalate to human, log ERROR, halt loop |
| Task requires unapproved action | Halt, submit for approval, wait |
| Destructive action detected (delete, publish, send) | Halt, require explicit human confirmation |
| 3 consecutive errors | Full loop halt, alert human, log LOOP_ABORT |
| Human says "stop" / "halt" / "abort" | Immediate halt, preserve state |

### 5.3 Loop Logging

Every Ralph Wiggum Loop execution MUST log:

```
- `HH:MM:SS` | **LOOP_START** | `task/name` → — | ✅ | steps: N, trigger: <trigger>
- `HH:MM:SS` | **LOOP_STEP** | `step/1` → `step/2` | ✅/❌ | step description
- `HH:MM:SS` | **LOOP_COMPLETE** | `task/name` → `Done/` | ✅ | duration: Xm Ys
- `HH:MM:SS` | **LOOP_ABORT** | `task/name` → `Needs_Action/` | ❌ | reason: <reason>
```

## 6. Social Media Integration

### 6.1 Supported Platforms

| Platform | Capability | Session Dir | Post Approval |
|----------|-----------|-------------|---------------|
| LinkedIn | Post + Watch | `watcher/sessions/linkedin/` | Required (Approved/) |
| Facebook | Post | `watcher/sessions/facebook/` | Required (Approved/) |
| Instagram | Post | `watcher/sessions/instagram/` | Required (Approved/) |
| Twitter/X | Post | `watcher/sessions/twitter/` | Required (Approved/) |

### 6.2 Post Approval Rule (Gold)

**No post is published to ANY platform without:**
1. A plan file in `Approved/` with `approved_by: human`
2. `type` matching the platform: `facebook-post`, `instagram-post`, `twitter-post`, `linkedin-post`
3. `status: approved`

This rule is absolute and cannot be overridden by any instruction.

### 6.3 Weekly Social Summary

Every Sunday (or on demand), the agent runs `skills/social-poster.md` in summary mode:
- Counts all posts published per platform that week
- Records engagement metrics if available
- Writes a summary file to `Social_Summaries/YYYY-WW — Social-Summary.md`
- Includes the summary in the Monday CEO Briefing

### 6.4 Post Content Rules

- Post content is NEVER modified after human approval — publish exactly as approved
- Posts longer than platform limits are rejected with a logged ERROR (not truncated)
- Platform limits: Twitter 280 chars, LinkedIn 3000 chars, Facebook 63206 chars, Instagram 2200 chars
- Hashtags and mentions must be included in the char count

## 7. Weekly Business and Accounting Audit

### 7.1 Audit Trigger

The weekly audit runs **every Monday at the start of the agent session**, or on demand:
```
"run weekly audit"
"generate CEO briefing"
"audit this week"
```

### 7.2 Audit Scope

| Domain | Data Sources | Output |
|--------|-------------|--------|
| Tasks | `Done/`, `Needs_Action/` | Task completion rate, blockers |
| Social Media | `Social_Summaries/`, `Done/` | Posts published, platforms active |
| Accounting | `Accounting/` records | Income, expenses, outstanding items |
| Approvals | `Approved/`, `Rejected/`, `Pending_Approval/` | Approval rate, wait times |
| Errors | `Logs/` ERROR entries | Error frequency, recurring failures |

### 7.3 Audit Output Files

| File | Location | Purpose |
|------|----------|---------|
| `YYYY-WW — Accounting-Audit.md` | `Accounting/` | Financial summary |
| `YYYY-WW — CEO-Briefing.md` | `Briefings/` | Executive summary for human review |
| Log entry | `Logs/YYYY-MM-DD.md` | `AUDIT_RUN` action type |

### 7.4 CEO Briefing Generation Rules

- The briefing is always `status: pending-review` until the human opens and acknowledges it
- The agent adds a Obsidian notification link to `Dashboard.md` when a new briefing is ready
- The briefing draws only from vault data — no external API calls for the briefing content itself
- Briefings are never deleted — archive to `Briefings/Archive/` after 90 days

## 8. MCP Server Architecture

Gold Tier introduces multiple MCP (Model Context Protocol) servers, each responsible for a specific action domain.

### 8.1 MCP Server Assignments

| MCP Server | Domain | Actions |
|-----------|--------|---------|
| `email_mcp.py` | Email | Read Gmail, draft replies, send (with approval) |
| Future: `social_mcp.py` | Social Media | Post to Facebook, Instagram, Twitter |
| Future: `calendar_mcp.py` | Scheduling | Read calendar, create events (with approval) |
| Future: `accounting_mcp.py` | Finance | Read transaction data, generate audit reports |

### 8.2 MCP Rules

- Each MCP server handles **one domain only** — no cross-domain actions in a single server
- All MCP servers extend `BaseWatcher` for consistent logging and vault path resolution
- MCP servers that write to external systems require an `Approved/` entry first
- MCP server failures trigger error recovery (see Section 9)

## 9. Error Recovery and Graceful Degradation

### 9.1 Error Classification

| Level | Trigger | Response |
|-------|---------|----------|
| **L1 — Transient** | Network timeout, API rate limit | Retry after 30s, max 2 retries |
| **L2 — Recoverable** | File not found, wrong format | Auto-fix if possible, else log WARNING |
| **L3 — Escalatable** | Auth failure, permission denied | Log ERROR, alert human, pause affected skill |
| **L4 — Critical** | Loop abort, data corruption | Log CRITICAL, halt all skills, alert human immediately |

### 9.2 Graceful Degradation

When a Gold-tier capability fails, the system **degrades gracefully**:

```
LinkedIn posting fails → Log ERROR → Move plan to Needs_Action/ for retry
Gmail API down → Log WARNING → Cache emails locally → Retry on next cycle
Weekly audit fails → Log ERROR → Generate partial briefing → Mark as incomplete
Ralph Wiggum Loop aborts → Save loop state → Resume from last checkpoint on retry
```

### 9.3 Recovery Log Format

```
- `HH:MM:SS` | **RECOVER** | `skill/name` → `retry/1` | ✅/❌ | error: <msg> | action: <recovery>
- `HH:MM:SS` | **ESCALATE** | `skill/name` → `Needs_Action/` | ❌ | escalated to human | reason: <msg>
```

## 12. Architecture Documentation

### 12.1 System Layers

```
┌─────────────────────────────────────────────────────┐
│  GOLD TIER — Cross-Domain Integration               │
│  Ralph Wiggum Loop | Social Posting | CEO Briefings │
├─────────────────────────────────────────────────────┤
│  SILVER TIER — Approval & Planning Layer            │
│  Plans/ | Pending_Approval/ | Approved/ | Rejected/ │
├─────────────────────────────────────────────────────┤
│  BRONZE TIER — Foundation                           │
│  Inbox/ | Needs_Action/ | Done/ | Logs/             │
├─────────────────────────────────────────────────────┤
│  WATCHER LAYER — Always-On Background Processes     │
│  filesystem | gmail | whatsapp | linkedin | twitter │
├─────────────────────────────────────────────────────┤
│  OBSIDIAN VAULT — File System as Single Source      │
│  AI_Employee_Vault/ — all state lives here          │
└─────────────────────────────────────────────────────┘
```

### 12.2 Data Flow — Social Posting

```
Human writes post idea → Plans/ (draft)
→ Agent formats post → submits to Pending_Approval/
→ Human approves → moves to Approved/
→ Agent reads Approved/ → publishes to platform
→ Moves plan to Done/YYYY-MM/
→ Updates Social_Summaries/ weekly entry
→ Included in next CEO Briefing
```

### 12.3 Data Flow — CEO Briefing

```
Weekly trigger (Monday) or "generate CEO briefing" command
→ Agent reads Done/ (tasks this week)
→ Agent reads Social_Summaries/ (posts this week)
→ Agent reads Accounting/ (financial records this week)
→ Agent reads Logs/ (errors, skill runs, approvals)
→ Agent writes Briefings/YYYY-WW — CEO-Briefing.md
→ Agent updates Dashboard.md with briefing link
→ Log entry: BRIEFING_GEN
```

### 12.4 Lessons Learned

| # | Lesson | Applied Where |
|---|--------|--------------|
| L-1 | Chrome profiles cannot be shared between concurrent Playwright processes | `linkedin_watcher.py` uses separate `sessions/linkedin-watcher/` |
| L-2 | PM2 browser scripts need a one-time manual login; sessions persist thereafter | `watcher/ecosystem.config.js` documented |
| L-3 | Skill files belong in `skills/` at vault root, not in `.claude/skills/` | All Gold skills in `skills/` |
| L-4 | Ralph Wiggum Loop must have hard-stop conditions to prevent runaway execution | See Section 5.2 |
| L-5 | Weekly briefings are only useful if generated consistently — schedule matters | Monday trigger in `skills/weekly-audit.md` |

## 16. Governance Rules

1. **All lower-tier constitutions remain authoritative** for their layers. Gold does not override Silver or Bronze.
2. **The agent re-reads all constitutions** (`Bronze-Constitution.md`, `Silver-Constitution.md`, `Gold-Constitution.md`) at the start of every session.
3. **The Ralph Wiggum Loop is never used for irreversible actions** without an `Approved/` entry.
4. **Skill files live in `skills/` only** — never in `.claude/skills/` or any other location.
5. **The approval gate applies to all external publishing** — no exceptions at any tier.
6. **Humans may edit this constitution** but must increment the version and log the change with action type `EDIT`.
